# CLAUDE.md

This file provides guidance to Claude Code (claude.ai/code) when working with code in this repository.

## What this repository is

A collection of decompiled HI-TECH C 3.09 (CP/M) tools, plus partial decompilations of HI-TECH v4.11 `libr` and `zas`. The original decompilation work for several tools was done by Andrey Nikitin; this repo restructures the code, verifies equivalence against the original CP/M binaries, fixes bugs, and adds a unified Windows (Visual Studio) and Linux (gcc) build. Also bundled: `hidump2` (enhanced dump), `zc` (C driver), `mkpat` (IDA Pro FLIRT pattern generator).

The code is intentionally written to remain buildable for CP/M targets where feasible, but cleanups for warnings may have introduced regressions on that path. See `Readme.md` for project history.

> **Ignore `AGENTS.md` in this repo.** Its contents describe an unrelated Z80 LLVM project (`/Users/ravn/z80/`) and do not apply here. `AGENT.md` is generic agent-workflow advice and is not specific to this codebase either.

## Repository layout

Each tool lives in its own top-level directory with the same internal shape:

- `<tool>/*.c`, `<tool>/*.h` â€” sources
- `<tool>/_appinfo.h` â€” defines `APP_NAME` (and optionally `APP_CONTRIBUTOR`); included via `hishared/appinfo.h`
- `<tool>/_version.h` â€” **autogenerated** from git history by `Scripts/getVersion.pl`; do not hand-edit
- `<tool>/<tool>.vcxproj` â€” Visual Studio project (referenced from `hitech.sln`)
- `Linux/<tool>/Makefile` â€” gcc build, includes `Linux/hi.mk`
- `<tool>/cpm/` or `<tool>/native/` (where present) â€” build configuration for cross-compiling back to CP/M via John Elliott's `zxcc` emulator

Tools: `cgen`, `cpp`, `cref`, `dehuff`, `dump`, `enhuff`, `hidump`, `hidump2`, `libr`, `libr4`, `link`, `mkpat`, `objtohex`, `optim`, `p1`, `zas`, `zas4`, `zc`. Shared code is in `hishared/`.

`runtime/` (top-level) holds the **Z80 target runtime** â€” 22 standard headers in `runtime/include80/`, plus runtime libraries (`LIBC.LIB`, `LIBF.LIB`, `LIBOVR.LIB`) and startup objects (`CRTCPM.OBJ`, `NRTCPM.OBJ`, `RRTCPM.OBJ`, `DRTCPM.OBJ`) in `runtime/lib80/`. These come from Tony Nicholson's `agn453/HI-TECH-Z80-C` consolidated distribution; provenance, license, and refresh procedure live in `runtime/README.md`. Source `runtime/env.sh` from the repo root to set `PATH` + `INCDIR80` + `LIBDIR80` and use `zc` end-to-end.

## Build commands

### Linux / WSL (gcc)

From `Linux/`:

```bash
make              # build all tools, installs binaries into Linux/Install/
make -C zas4      # build a single tool
make clean        # remove .o files
make distclean    # also remove built binaries
make rebuild      # distclean + build
make publish      # force version refresh, then build
make install      # copy to Linux/Install/ (also done by `all`)
```

`Linux/hi.mk` is the common include used by every per-tool Makefile. Default flags: `gcc -O3 -Wall -I<srcdir> -I<root>/hishared`. The build copies the resulting binary into `Linux/Install/` automatically.

### Health check

Run `Scripts/check.sh` from anywhere to verify the working tree is in shape: all 18 host tools built, runtime/ tree intact, end-to-end `zc hello.c` produces a runnable `.com`, RunCPM executes it (if the emulator is available), git position vs `origin`/`upstream/main` reasonable, and `ogdenpm/hitech#5` PR status. Designed for a quick "am I good to start?" check when sitting back down at the project. Exits non-zero on any failure.

### Docker image (`ghcr.io/ravn/hitech:latest`)

`Dockerfile` + `.github/workflows/container.yml` build a `linux/arm64` container with all 18 tools and the vendored runtime, published to GitHub Container Registry on every main push. **Status: the image-build mechanism works (tools run, `-V` reports versions correctly), but the toolchain inside currently can't compile real source â€” `cgen` rejects p1's intermediate output with `Bad int. code` on Linux/glibc.** Same root cause as the CI smoke-test SIGABRT and tracked alongside it in `tasks/todo.md`. Fixing the underlying p1 / decompiled-code bug would unbreak both at once.

### Windows (Visual Studio)

Open `hitech.sln` in Visual Studio 2022 and build. `hishared/Cbuild.props` and `hishared/Version.props` are shared props; `hishared/hishared.vcxitems` is the shared-source items project referenced by every tool.

### CP/M cross-build (selected tools only)

`zas/cpm/`, `p1/native/`, `cgen/native/` etc. contain Makefiles that drive `zxc`/`zxas`/`zxcc` to rebuild the original CP/M `.com` binaries. Requires `zxcc` and the HI-TECH cross-compiler tools on `PATH`. `cgen/native/` first runs `unpack.pl` to extract Huffman-compressed sources from `cgen/nikitin/*.HUF`.

## Versioning system (important to understand)

Every tool reports a version of the form `yyyy.mm.dd.nn` derived from git, where `yyyy.mm.dd` is the date of the last commit affecting that tool's directory and `nn` is the release-tag sequence. This is enforced by:

- `Scripts/getVersion.pl` (and `getVersion.cmd` on Windows) â€” scans the tool's directory in git, writes `_version.h` containing `#define GIT_VERSION "..."`. Invoked from `Linux/hi.mk` as `chkversion` (`-w`, write only if changed) or `mkversion` (`-W`, always rewrite to force a rebuild).
- An optional `version.in` config file overrides defaults.
- `hishared/_version.c` + `hishared/showVersion.h` implement the `-v` / `-V` CLI flags. Every tool's `main()` should call `CHK_SHOW_VERSION(argc, argv)` early. `-v` prints product + git version; `-V` adds description, build timestamp, contributors, and support email pulled from `<tool>/_appinfo.h` and `hishared/appinfo.h`.

A trailing `+` on the version means uncommitted changes; a trailing `?` means git was unavailable. See `Scripts/versionTools.pdf` for full details.

When a build can't find git, `_version.h` is preserved with `?` appended â€” committed `_version.h` files are intentional fallbacks, not stale artifacts.

## Code style

`.clang-format` is set up for the project: 4-space indent (tab width 4), 100-column limit, CRLF line endings, aligned consecutive assignments and macros, no short-form blocks/ifs/loops on a single line, `IndentCaseBlocks: true`. Run clang-format before submitting changes.

`.gitattributes` enforces line-ending normalization; do not fight it.

## Working on decompiled code

Most `.c` files in `cgen/`, `p1/`, `zas/`, `optim/`, `link/`, `libr/` are the result of decompiling CP/M binaries. Expect:

- Symbol names like `sub_6AD0`, `array_AE13`, `byte_AFFA` from the disassembler â€” these are progressively being renamed as semantics are recovered. Comments next to globals often list the original calling functions.
- `#if 0` blocks marking unused functions/variables that exist in the binary but were never reached.
- Comparison spreadsheets (`p1codecmp.xlsm`, `zascodecmp.xlsm`) tracking decompilation parity against the original.
- Bug-fix changes versus the original CP/M code are intentional â€” see `zas4/readme.md` for an example list. Don't "restore" old behavior without checking the commit log and that file.

When fixing a bug, verify against the original CP/M binary behavior where the change might affect the decompilation-fidelity goal.
