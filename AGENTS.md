For repository orientation and build commands see `CLAUDE.md`. For project philosophy see `AGENT.md`.

## Project at a glance

- **What it is:** Decompiled HI-TECH C 3.09 tools (`cgen`, `cpp`, `link`, `optim`, `p1`, `zas`, `libr`, тАж) plus partial v4.11 decompilations (`libr4`, `zas4`) and a few utilities (`hidump`, `hidump2`, `mkpat`, `zc`, `dehuff`, `enhuff`, `objtohex`, `cref`, `dump`).
- **Targets:** Modern hosts (Windows / Linux / macOS) **and** the original CP/M target (via `zxcc` in `<tool>/cpm/` or `<tool>/native/`).
- **Goal:** Provide a working HI-TECH C compiler on modern systems while preserving the ability to rebuild the original CP/M `.com` binaries from the decompiled sources.
- **Source state:** Many functions still carry placeholder names (`sub_6AD0`, `array_AE13`, `byte_AFFA`) from the disassembler. Renaming and semantic recovery is incremental.

## Critical design constraint: dual-target portability

The same C source must compile cleanly for:

1. **Modern host** (Visual Studio, gcc on Linux/WSL, Apple clang on macOS) тАФ for daily use as a Z80 cross-compiler.
2. **Original CP/M** (via `zxcc` running HI-TECH's own `zxc`/`zxas`) тАФ to regenerate the binaries the decompilation came from and verify equivalence.

The project's canonical macro for the CP/M target is **`CPM`** (positive test). Use `#ifdef CPM` to gate CP/M-only code; use `#ifndef CPM` for modern-host-only code. Examples: `dehuff/dehuff.c:29`, `libr/libr.c:726`, `cgen/cgen.h:77`, `hishared/showVersion.h:9`.

**Do not introduce new negative tests** like `#if !defined(unix) && !defined(_WIN32)`. The `unix` macro is predefined by GCC on Linux but **not** by Apple clang on macOS, so such tests silently misbehave on macOS. If you see one (e.g. `enhuff/enhuff.c:58`, `cpp/cpp.c:1895`), convert it to `#ifdef CPM` / `#ifndef CPM`.

## Repository layout

Each tool lives in its own top-level directory with the same shape:

- `<tool>/*.c`, `<tool>/*.h` тАФ sources
- `<tool>/_appinfo.h` тАФ defines `APP_NAME` (and optionally `APP_CONTRIBUTOR`)
- `<tool>/_version.h` тАФ **autogenerated** by `Scripts/getVersion.pl`; do not hand-edit
- `<tool>/<tool>.vcxproj` тАФ Visual Studio project (referenced from `hitech.sln`)
- `Linux/<tool>/Makefile` тАФ gcc/clang build, includes `Linux/hi.mk`
- `<tool>/cpm/` or `<tool>/native/` (where present) тАФ CP/M cross-build via `zxcc`

Shared code lives in `hishared/` (`showVersion.h`, `_version.c`, `appinfo.h`, `Cbuild.props`, `Version.props`).

The **Z80 target runtime** is vendored in `runtime/` at the repo root (headers in `include80/`, libraries and startup objects in `lib80/`). It is sourced from Tony Nicholson's `agn453/HI-TECH-Z80-C` consolidated distribution; see `runtime/README.md` for provenance and refresh instructions, and `runtime/LICENSE.HITECH` for the freeware grant. `source runtime/env.sh` from the repo root to point a host-built `zc` at the headers, libraries, and the locally built host tools.

## Build workflows

### Linux / macOS (gcc / Apple clang)

From repo root:

```bash
cd Linux
make                      # build everything тЖТ Linux/Install/
make -C zas4              # build a single tool
make -C zas4 rebuild      # distclean + build
make clean                # remove .o
make distclean            # also remove built binaries
make publish              # force version refresh, then build
```

`Linux/hi.mk` is the common include. Default flags: `gcc -O3 -Wall -I<srcdir> -I<root>/hishared`. On macOS with Xcode Command Line Tools installed, both `/usr/bin/cc` and `/usr/bin/gcc` resolve to Apple clang (verified on this machine: clang 21.0.0, arm64-apple-darwin25.4.0), so the unmodified Linux Makefile builds via clang on macOS.

### Windows (Visual Studio)

Open `hitech.sln` in VS 2022. `hishared/Cbuild.props` and `hishared/Version.props` are shared props; `hishared/hishared.vcxitems` is the shared-source items project referenced by every tool.

### CP/M cross-build (selected tools)

`zas/cpm/`, `p1/native/`, `cgen/native/`, etc. drive `zxc`/`zxas`/`zxcc` to rebuild the original CP/M `.com` binaries. Requires `zxcc` and the HI-TECH cross-tools on `PATH`. `cgen/native/` first runs `unpack.pl` to extract Huffman-compressed sources from `cgen/nikitin/*.HUF`. These paths exist to verify decompilation parity тАФ do not casually break them.

### Verifying a single tool builds on macOS

```bash
make -C Linux/<tool> rebuild
./Linux/Install/<tool> -V         # prints product, git version, build time, contributors
```

If `-V` shows `+` after the version, you have uncommitted changes. If it shows `?`, git was unavailable during build.

### Health check (`Scripts/check.sh`)

A repository-wide smoke test for "am I good to start?" sessions. Run it from anywhere; it cd's into the repo root and runs:

1. **Host build** тАФ checks every name in the 18-tool list has a binary in `Linux/Install/`.
2. **Runtime tree** тАФ counts `*.H` files in `runtime/include80/` (expects 22) and `*.LIB`/`*.OBJ` in `runtime/lib80/` (expects 7).
3. **Smoke test** тАФ compiles a tiny `hello.c` with the local `zc` against the runtime tree and confirms the resulting `.com` carries the HI-TECH signature.
4. **RunCPM** тАФ if `/Users/ravn/git/RunCPM/RunCPM/RunCPM` exists (override with `RUNCPM_DIR`), actually executes the `.com` and checks the printed line.
5. **Git state** тАФ branch, ahead/behind `origin` and `upstream/main`, dirty working tree, local-only branches with no upstream tracking.
6. **PR #5** тАФ if `gh` is installed and authenticated, prints the merged/open state of `ogdenpm/hitech#5` and the comment count.

Each step prints PASS / FAIL / SKIP. Exits non-zero on any FAIL. Designed to be run when you sit back down at the project, not on every commit.

## Versioning

`Scripts/getVersion.pl` reads git history for the tool's directory and writes `_version.h` containing `#define GIT_VERSION "yyyy.mm.dd.nn"`. Invoked from `Linux/hi.mk`:

- `chkversion` (`-w`) тАФ write only if the value changed
- `mkversion` (`-W`) тАФ always rewrite to force a rebuild

Trailing `+` = uncommitted changes; trailing `?` = git unavailable. `hishared/_version.c` and `hishared/showVersion.h` implement the `-v` / `-V` flags. Every tool's `main()` calls `CHK_SHOW_VERSION(argc, argv)` early. See `Scripts/versionTools.pdf` for full details.

## Key conventions

### Decompilation parity is a goal, not a chore

`Readme.md` states the explicit goal of verifying against the original CP/M binaries. `zas4/readme.md` lists bug fixes made *relative to* the original тАФ those are deliberate, not regressions. Before "fixing" something that looks wrong:

1. Check `git log` for the file тАФ was it recently changed for parity reasons?
2. Check the tool's `readme.md` (where present) for an explicit deviation list.
3. If still unsure, ask before changing.

The `*codecmp.xlsm` spreadsheets (`p1/p1codecmp.xlsm`, `zas/zascodecmp.xlsm`) track parity and are worth consulting for those tools.

### Placeholder identifiers

Names like `sub_6AD0`, `array_AE13`, `byte_AFFA`, `word_AFF8` are leftovers from the disassembler. Comments next to globals often list which functions reference them тАФ useful when renaming. Renaming is welcome but must preserve behaviour exactly; do not combine renames with logic changes in the same commit.

### `#if 0` blocks

Common in decompiled code тАФ they mark functions/variables that exist in the binary but were never reached at runtime. Leaving them in is intentional documentation. Don't delete unless you have reason to believe they were truly dead in the original code as well.

### Code style

`.clang-format` exists at the repo root: 4-space indent (tab width 4), 100-column limit, **CRLF line endings** (`UseCRLF: true`), aligned consecutive assignments and macros, no short-form blocks/ifs/loops on a single line, `IndentCaseBlocks: true`. `.gitattributes` enforces line-ending normalization (`* text=auto`). Whether contributors actually run `clang-format -i` before commits is not documented anywhere I can see тАФ match the surrounding style of the file you're editing if in doubt.

## Known portability gaps (macOS, as of 2026-05-02)

Verified by reproducing the build on `/usr/bin/cc` (Apple clang 21.0.0, `arm64-apple-darwin25.4.0`):

- **`enhuff/enhuff.c:58`** тАФ `#if !defined(unix) && !defined(_WIN32)` gates a CP/M-only `_getargs`/`_argc_` block. Apple clang does not predefine `unix` (GCC on Linux does), so the CP/M-only block is compiled and references undeclared symbols. Build error reproduced. Suggested fix: convert to `#if CPM`, matching the pattern in `dehuff/dehuff.c:29`. *(Fix not yet applied / not yet tested under the CP/M cross-build.)*

- **`cpp/cpp.c:1895`** тАФ `#if defined(unix) || defined(_WIN32)` skips `dirs[0] = dirnams[0] = "."` initialisation on macOS. Compiles cleanly. *Suspected but unverified* runtime impact: `dirs[]` is a `STATIC` array so the slot reads as NULL, which would cause the include-search loop at `cpp.c:998` (`for (dirp = dirs + inctype; *dirp; ++dirp)`) to terminate immediately and miss the source's own directory when no `-I.` flag is passed. **Not reproduced** тАФ needs an actual cpp invocation to confirm.

If you hit a new portability issue: prefer the project's `CPM` macro convention over adding `__APPLE__` to existing negative tests. If you "fix" something on the modern-host side, also build the corresponding `<tool>/cpm/` target (where it exists) to confirm CP/M parity is preserved.

## Common AI-agent workflows

### Investigating a build break on a new host

1. `cd Linux && make 2>&1 | tee /tmp/build.log` тАФ let it run to first failure.
2. Identify the first failing translation unit; read its source for `#if`/`#ifdef` gates around the failing identifiers.
3. Check whether other tools handle the same construct correctly (`grep -rn "<symbol>"`) тАФ usually one tool already has the right fix that can be replicated.
4. Verify the fix doesn't break the CP/M cross-build by reading the `<tool>/cpm/` or `<tool>/native/` Makefile and the same source under `#ifdef CPM`.

### Renaming a placeholder symbol (`sub_XXXX`, `array_XXXX`)

1. `grep -rn "<old_name>" <tool>/` тАФ verify it is local to one tool (most are).
2. Read the comments next to the declaration тАФ they often list call sites and original semantics.
3. Rename via `Edit` with `replace_all` on each affected file.
4. Build with `make -C Linux/<tool> rebuild` and run a smoke test if the tool has one.
5. Commit with a message that says "rename only тАФ no behaviour change".

### Adding a new tool / new source file

1. Add the file to the tool's `<tool>.vcxproj` (Visual Studio) and to `OBJS` in `Linux/<tool>/Makefile` (gcc).
2. If new headers are involved, add the dependency line to the bottom of `Linux/<tool>/Makefile` (see `Linux/zas4/Makefile` for pattern).
3. If the file uses any host-specific code, gate with `#ifdef CPM` / `#ifndef CPM`, never `unix`/`__APPLE__`.
